# Kubernetes Monitoring Platform

## Overview

This project implements a monitoring and observability platform for the Kubernetes homelab.

The monitoring stack collects metrics from the Kubernetes cluster, nodes, Pods, containers and control-plane components. Grafana uses these metrics to display cluster health and resource utilization through dashboards.

The platform was deployed using the `kube-prometheus-stack` Helm chart.

---

## Architecture

```text
Kubernetes Cluster
        |
        +-- kube-state-metrics
        |       |
        |       +-- Kubernetes object metrics
        |
        +-- Node Exporter
        |       |
        |       +-- CPU, memory, disk and network metrics
        |
        +-- Prometheus
        |       |
        |       +-- Metric collection and storage
        |
        +-- Alertmanager
        |       |
        |       +-- Alert processing
        |
        +-- Grafana
                |
                +-- Dashboards and visualization
```

---

## Components

| Component | Purpose |
|---|---|
| Prometheus | Collects and stores time-series metrics |
| Grafana | Visualizes metrics through dashboards |
| Alertmanager | Processes alerts generated by Prometheus |
| Node Exporter | Exposes Linux node metrics |
| kube-state-metrics | Exposes Kubernetes object metrics |
| Prometheus Operator | Manages Prometheus resources in Kubernetes |

## Helm Deployment

The Prometheus Community Helm repository is used to deploy the monitoring stack.

```bash
helm repo add prometheus-community \
https://prometheus-community.github.io/helm-charts

helm repo update
```

The installed Helm releases can be verified with:

```bash
helm list -A
```

The homelab-specific configuration is stored in:

```text
monitoring/prometheus/helm/custom-values.yaml
```

Example installation command:

```bash
helm upgrade --install monitoring \
prometheus-community/kube-prometheus-stack \
--namespace default \
-f monitoring/prometheus/helm/custom-values.yaml
```

> The namespace and release name must match the currently deployed Helm release.

## Prometheus Access

Prometheus can be accessed temporarily using port forwarding:

```bash
kubectl port-forward \
service/monitoring-kube-prometheus-prometheus \
-n default \
9091:9090 \
--address 0.0.0.0
```

Access:

```text
http://192.168.100.30:9091
```

The port-forward command must remain running while Prometheus is being accessed.

## Namespace Dashboard

A namespace variable allows the dashboard to filter Kubernetes workloads dynamically.

Example variable configuration:

```text
Name: namespace
Query type: Label values
Label: namespace
Metric: kube_pod_info
```

Example PromQL query:

```promql
count(
  kube_pod_status_phase{
    namespace=~"$namespace",
    phase="Running"
  } == 1
)
```

---

## Useful PromQL Queries

### Prometheus targets currently up

```promql
count(up == 1)
```

### Ready nodes

```promql
sum(kube_node_status_condition{condition="Ready",status="true"})
```

### Running Pods

```promql
count(kube_pod_status_phase{phase="Running"} == 1)
```

### Pending Pods

```promql
count(kube_pod_status_phase{phase="Pending"} == 1)
```

### Failed Pods

```promql
count(kube_pod_status_phase{phase="Failed"} == 1)
```

### Pod restarts during the last hour

```promql
sum(increase(kube_pod_container_status_restarts_total[1h]))
```

### Namespace CPU usage

```promql
sum(
  rate(
    container_cpu_usage_seconds_total{
      namespace=~"$namespace",
      container!="",
      image!=""
    }[5m]
  )
)
```

### Namespace memory usage in MiB

```promql
sum(
  container_memory_working_set_bytes{
    namespace=~"$namespace",
    container!="",
    image!=""
  }
) / 1024 / 1024
```

## Persistent Storage

Prometheus uses a dedicated PersistentVolume.

Manifest:

```text
kubernetes/storage/persistent-volumes/prometheus-pv.yaml
```

The storage directory on the Kubernetes node is:

```text
/data/kubernetes/prometheus
```

The directory permissions must allow the Prometheus container to write data.

Example:

```bash
sudo chown -R 1000:2000 /data/kubernetes/prometheus
sudo chmod -R 775 /data/kubernetes/prometheus
```

## Troubleshooting

### Prometheus Pod in CrashLoopBackOff

Check the Pod:

```bash
kubectl get pods -n default
```

Inspect events:

```bash
kubectl describe pod \
prometheus-monitoring-kube-prometheus-prometheus-0 \
-n default
```

Check Prometheus logs:

```bash
kubectl logs \
prometheus-monitoring-kube-prometheus-prometheus-0 \
-n default \
-c prometheus
```

A previously encountered issue was:

```text
opening query log file:
open /prometheus/queries.active:
permission denied
```

The issue was resolved by correcting ownership and permissions on the host directory used by the PersistentVolume.

### Port already in use

Check the process using the port:

```bash
ss -lntp | grep 9090
```

Use another local port if necessary:

```bash
kubectl port-forward \
service/monitoring-kube-prometheus-prometheus \
-n default \
9091:9090 \
--address 0.0.0.0
```

### Grafana shows no data

Validate that Prometheus is configured as a Grafana data source and test the following query:

```promql
up
```

Also verify:

```bash
kubectl get pods -n default
kubectl get svc -n default
```

## Lessons Learned

- Prometheus stores Kubernetes and infrastructure metrics as time series.
- Grafana queries Prometheus using PromQL.
- Node Exporter provides host-level metrics.
- kube-state-metrics provides Kubernetes object metrics.
- Persistent storage prevents Prometheus data loss after Pod recreation.
- Helm values provide repeatable monitoring deployments.
- Dashboard variables make dashboards reusable across namespaces.
- Logs, events and persistent-volume permissions are essential when troubleshooting monitoring Pods.
